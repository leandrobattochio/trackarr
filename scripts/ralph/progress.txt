## Codebase Patterns
- YAML plugin model classes live in `TrackerStats.Domain/Plugins/Yaml/` namespace
- Runtime plugin loading resolves `Plugins:Directory` relative to the API content root unless the config value is already absolute
- Disk YAML definitions load first and DB-stored JSON definitions override them by `pluginId`; `/api/plugins` surfaces the merged source as `disk` or `database`
- `TrackerPluginRegistry` reloads definitions through `IYamlPluginDefinitionLoader` on each query so plugin CRUD changes are visible immediately without restarting the API
- Frontend plugin detail fetches should use a text response helper: `GET /api/plugins/{pluginId}` returns raw YAML, not JSON
- Plugin editor mutations should invalidate both the catalog query and the per-plugin definition query so save/delete actions show up immediately in the sidebar and editor
- Domain project has no external dependencies — plain C# classes with `required` keyword for mandatory fields
- Existing plugin domain types are in `TrackerStats.Domain/Plugins/` (ITrackerPlugin, TrackerStats, TrackerFetchResult, etc.)
- Use `dotnet build src/TrackerStats.Domain/TrackerStats.Domain.csproj --no-restore` for quick typecheck of Domain layer
- YAML plugin files go in `backend/plugins/` directory — use `python3 -c "import yaml; yaml.safe_load(open('file.yaml'))"` to validate
- Mapping syntax: `steps.<stepName>.<varName>` for extracted values, `fields.<fieldName>` for config, arithmetic for computed
- Unit3D plugins share identical structure: single GET to `api/user?api_token={{apiKey}}`, JSON response, HTTP-only auth failure (401/403)
- EF Core migration: `dotnet ef migrations add <Name> -p src/TrackerStats.Infrastructure -s src/TrackerStats.Api` (requires global dotnet-ef tool)
- DI registration pattern: interface in Domain, implementation in Infrastructure, `AddScoped` in Program.cs
- Entity pattern: simple POCO in `Domain/Entities/`, Fluent API config in `AppDbContext.OnModelCreating()`
- YAML engine: interface `IYamlPluginEngine` in Domain, implementation `YamlPluginEngine` in `Infrastructure/Plugins/Yaml/`
- YAML plugin registry must clone plugin prototypes, because multiple plugin IDs can now share the same `YamlTrackerPlugin` implementation type
- Template interpolation: `InterpolateFieldValues` (no URL encoding, for headers/baseUrl) vs `InterpolateTemplate` (URL-encodes, for step URLs)
- Auth failure detection checks HTTP status codes first, then HTML patterns — returns early on first match
- `YamlPluginEngine.ExtractValues` and `BuildResult` are `protected virtual` — US-007 overrides or extends them

- Engine capabilities:

  - Configures HttpClient from definition (baseUrl, headers, cookies with {{field}} interpolation)
  - Executes ordered request steps sequentially (GET/POST/PUT/DELETE)
  - Template interpolation: {{fieldName}} for config values, {{steps.stepName.varName}} for extracted step values
  - Auth failure detection: HTTP status codes list + HTML content pattern matching
  - Two interpolation modes: `InterpolateFieldValues` (raw, for headers) and `InterpolateTemplate` (URL-encoded, for step URLs)
  - `ExtractValues` and `BuildResult` are `protected virtual` — placeholders for US-007 to implement parsing/transformations
- **Learnings for future iterations:**
  - Headers should NOT be URL-encoded (e.g., Cookie values, Accept header) — use `InterpolateFieldValues`
  - Step URLs should URL-encode interpolated values — use `InterpolateTemplate`
  - `partial class` with `[GeneratedRegex]` works well for compile-time regex in .NET 10
  - Cookie header needs `TryAddWithoutValidation` (not `Add`) because HttpClient validates standard headers strictly
  - User-Agent needs `UserAgent.ParseAdd()` — direct header add fails validation
